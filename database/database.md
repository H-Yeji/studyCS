### **✅ 2. RDB와 NoSQL의 차이에 대해 설명해 주세요.**
```
🗯️ RDB는 SQL 쿼리를 사용해 데이터를 조작하는 관계형 데이터베이스로, 데이터를 테이블의 형태로 저장합니다.
반면, NoSQL은 SQL을 사용하지 않는 비관계형 데이터베이스로, 유연한 스키마와 다양한 데이터 모델을 지원합니다. 
따라서 RDB는 구조화된 데이터를 처리하는 데 강점을 가지며, NoSQL은 빠르게 변하는 반구조화 데이터를 처리하는 데 강점을 가집니다.
```

- **NoSQL의 강점과, 약점이 무엇인가요?**
```
🗯️ NoSQL의 주요 강점은 유연한 데이터 구조를 제공하고, 수평 확장이 용이해 대규모 데이터를 처리하는 데 적합하다는 점입니다.
또한 비정형 및 반정형 데이터를 효율적으로 처리할 수 있고, 분산 환경을 지원하여 높은 확장성을 제공합니다.

반면, 약점은 데이터 중복이 발생할 수 있고, RDB와 다르게 명확한 데이터 구조를 보장하지 않기 때문에
데이터의 일관성을 유지하는 데 어려움이 있을 수 있다는 점입니다.
```
  
- **RDB의 어떠한 특징 때문에 NoSQL에 비해 부하가 많이 걸릴 "수" 있을까요? (주의: 무조건 NoSQL이 RDB 보다 빠르다라고 생각하면 큰일 납니다!)**
```
🗯️ RDB는 고정된 스키마와 정규화된 데이터 모델을 기반으로 합니다.
그렇기 때문에 데이터의 양이 증가하거나, Join 연산 같이 복잡한 연산, 복잡한 쿼리가 수행될 때 부하가 커질 수 있습니다.
또한 RDB는 수직  확장에 의존하는 경우가 많아, 서버의 성능을 높이는 방식으로만 확장하므로 대규모 트래픽을 처리하는 데 한계가 있을 수 있습니다.
따라서 대규모 데이터나 고속 트랜잭션 처리를 요구하는 시스템에서는 NoSQL에 비해 부하가 많이 걸릴 수 있습니다. 
```
  
- **NoSQL을 활용한 경험이 있나요? 있다면, 왜 RDB를 선택하지 않고 해당 DB를 선택했는지 설명해 주세요.**
```
🗯️ 직접 구현한 적은 없지만, 이전 프로젝트에서 같은 팀원이 Redis를 활용한 경험이 있습니다.
메인 페이지와 같이 자주 조회되지만 데이터 변경이 적은 페이지에 대해 캐시를 적용했는데,
먼저 Redis에서 데이터를 조회하고, 만약 데이터가 없다면 데이터베이스에서 가져와 Redis에 저장했습니다.
그리고 그 이후 같은 요청이 들어오면 Redis에서 빠르게 데이터를 제공하는 방식을 사용하여
RDB를 사용할 때보다 부하를 줄이고 조회 성능을 향상시킬 수 있었습니다. 
```

### **✅ 3. 트랜잭션이 무엇이고, 3.ACID 원칙에 대해 설명해 주세요.**
```
🗯️ 트랜잭션은  select, insert, delete, update와 같은 sql 질의어를 통해 데이터베이스의 상태를 변화시키기 위해 수행하는 작업을 말합니다.
예를 들어 게시판을 작성 기능을 구현한다고 가정하면, 글을 작성하고 완료하면 해당 글로 리다이렉트 되는 것을 볼 수 있습니다.
이것을 질의어로 나타내면 insert문으로 글을 데이터베이스에 저장하고 select문으로 해당 글을 조회할 수 있는데,
이 때 insert문과 select 문은 하나의 트랜잭션으로 이루어지는 것입니다. 

🗯️ 트랜잭션의 ACID 원칙은 원자성, 일관성, 독립성, 지속성을 말합니다. 
먼저 원자성은 트랜잭션이 진행되면 데이터베이스에 모두 커밋되어 반영되든지, 혹은 모두 롤백되어 반영되지 않아야 하는 것을 말합니다. 
두 번째 일관성은 트랜잭션의 작업 처리 결과가 항상 일관성 있는 데이터베이스 상태를 유지해야 한다는 것입니다. 
세 번째 독립성은 둘 이상의 트랜잭션이 진행되고 있을 경우, 어떤 하나의 트랜잭션이라도 다른 트랜잭션의 연산에 끼어들 수 없는 것을 의미합니다.
마지막으로 지속성은 트랜잭션의 작업이 성공적으로 완료되면, 그 결과는 영구적으로 반영되어야 함을 의미합니다.
```

- **ACID 원칙 중, Durability를 DBMS는 어떻게 보장하나요?**
```
🗯️ 로그 파일을 통해 보장할 수 있습니다.
데이터베이스는 트랜잭션이 실행되는 동안 발생하는 모든 변경 사항을 로그 파일에 기록합니다.
그 후에 실제 데이터베이스에 적용되는데, 시스템에 장애가 발생하면 이 로그를 사용해 데이터베이스를 이전의 상태로 복구할 수 있습니다.
```

- **트랜잭션을 사용해 본 경험이 있나요? 어떤 경우에 사용할 수 있나요?**
```
🗯️ 기본적인 qna 게시판을 작성하고 수정하는 과정이나, 호텔 예약 서비스를 구현할 때 예약 과정에 트랜잭션을 사용했습니다.
호텔 예약을 진행할 때는 인원 수와, 날짜, 그리고 원하는 방의 타입을 선택하는데
이 선택 과정에서 하나라도 문제가 발생하면 예약이 진행되지 않고 모두 롤백 될 수 있도록 트랜잭션을 사용하였습니다. 
```
  
- **읽기에는 트랜잭션을 걸지 않아도 될까요?**
```
🗯️ 보통은 읽기 작업에서 트랜잭션 없이도 수행할 수 있지만, 일관성과 격리성을 보장하기 위해 트랜잭션을 거는 경우도 있습니다.
예를 들어 read committed 수준에서는 읽는 동안 다른 트랜잭션이 데이터를 수정할 수 있어 비일관성이 발생할 수 있는데,
이런 경우 트랜잭션을 걸어 비일관성을 방지할 수 있습니다. 
```

### **✅ 7. 정규화가 무엇인가요?**

```
🗯️ 정규화란 이상현상이 있는 릴레이션을 분해해 이상현상을 없애는 과정입니다.
데이터베이스를 잘못 설계하면 불필요한 데이터의 중복이 발생하여 데이터의 삽입, 수정, 삭제 연산을 수행할 때 부작용이 발생할 수 있습니다. 
이런 이상현상을 제거하면서 데이터베이스를 올바르게 설계해나가는 과정을 ‘정규화’라고 합니다. 
```

- **정규화를 하지 않을 경우, 발생할 수 있는 이상현상에 대해 설명해 주세요.**
``` 
🗯️ 정규화를 하지 않을경우 삽입 이상, 갱신 이상, 삭제 이상이 발생할 수 있습니다.
먼저 삽입 이상은 새 데이터를 삽입하기 위해 불필요한 데이터도 함께 삽입해야 하는 문제입니다. 
두번째 갱신 이상은 중복 튜플 중 일부만 변경하여 데이터가 불일치하게 되는 모순의 문제입니다.
마지막으로 삭제 이상은 튜플을 삭제할 때 꼭 필요한 데이터까지 함께 삭제되는 데이터 손실의 문제입니다.
```
- **각 정규화에 대해, 그 정규화가 진행되기 전/후의 테이블의 변화에 대해 설명해 주세요.**
```
🗯️ 제 1정규화는 테이블의 컬럼이 원자값을 갖도록 테이블을 분해하는 것입니다.
제 1정규화를 거치면 한 컬럼에 두 개 이상의 데이터가 존재했던 것에서 단일 값으로 변경됩니다.

제 2정규화는 제 1정규화를 거친 후 부분 함수 종속을 없애는 것입니다.
부분 함수 종속이란 기본키가 여러 컬럼일 때, 기본키의 일부에만 의존하는 컬럼이 있는 경우를 말합니다.
이 과정을 거치면 기본키를 분리하여 각 컬럼이 기본키 전체에 의존하도록 완전 함수 종속으로 변경됩니다.

제 3정규화는 제 2정규화를 거친 후 이행적 종속을 없애도록 분해되는 것입니다.
이 과정을 거친 후 모든 컬럼은 기본키에 이행적 함수 종속이 되지 않고 기본키에만 의존하도록 변경됩니다.

보이스-코드 정규화는 제 3정규화를 진행한 테이블에 대해 모든 결정자가 후보키가 되도록 테이블을 분해하는 것입니다.
이 과정을 거치면 모든 결정자가 후보키가 되어 테이블에서 비정상적인 종속 관계를 제거합니다.

제 4정규화는 하나의 컬럼이 여러 값을 가지는 다치 종속을 제거하는 것입니다.
다치 종속을 없애기 위해 다치 종속 관계를 별도의 테이블로 분리합니다.

제 5정규화는 조인 종속을 제거하는 것입니다.
이 과정을 거치면 후보키를 통하지 않는 조인 종속이 제거됩니다.
```
- **정규화가 무조건 좋은가요? 그렇지 않다면, 어떤 상황에서 역정규화를 하는게 좋은지 설명해 주세요.**
```
🗯️ 정규화가 항상 좋은 것은 아닙니다.
정규화로 인해 테이블이 분리되면 여러 테이블을 조인해야 하는 경우가 많아져 쿼리 성능이 저하될 수 있습니다. 
따라서 데이터의 조회가 자주 일어나는 경우나, 복잡한 쿼리를 사용하고 빈번하게 조인이 필요한 경우에는 역정규화를 하는 것이 좋을 수 있습니다.
```

```
🗯️ 스키마는 외부 스키마, 개념 스키마, 내부 스키마 3계층 구조로 나뉩니다.

사용자나 응용 프로그램이 각자의 관점에 따라 동일한 데이터를 다르게 정의할 수 있도록 허용하고,

개념 스키마는 개체 간의 관계, 무결성 제약 조건, 접근 권한 등을 포함해 데이터베이스의 주요 구조를 규정합니다.
데이터베이스의 전체적인 논리적 구조를 정의하고, 데이터베이스 관리자에 의해 관리됩니다.
또한 각 데이터베이스는 1개의 개념 스키마만 존재합니다.

내부 스키마는 데이터베이스의 물리적 구조를 정의한 것입니다.
데이터의 내부 저장 방식, 레코드 형식, 저장 순서 등을 다루며 실제로 데이터를 어떻게 저장하고 관리할지 정의합니다.
```
